# Implementation Plan: Igniter Installer

**Generated:** 2026-04-25

## Goal

Implement an Igniter-driven Mix task for the `phoenix_lite` package that, when
run against a freshly generated Phoenix project, applies all codemods needed
for the project to boot under `MIX_ENV=prod` with zero further manual edits â€”
primarily by replacing `config/runtime.exs` with a pre-configured templated
version (SSL off, host `0.0.0.0`, auto-generated secret key base) while
preserving a backup of the original.

## Scope

### In scope

- A Mix task module (`Mix.Tasks.PhoenixLite.Install`) discoverable by `mix igniter.install phoenix_lite` and `mix igniter.new ... --install phoenix_lite`.
- A templated `runtime.exs` shipped inside the `phoenix_lite` package, applied to the host project at install time.
- A backup mechanism for the host project's original `runtime.exs`.
- Idempotent re-runs (second run is a no-op or otherwise non-destructive).
- A summary of files written/backed up surfaced through Igniter's normal notice channel.
- A modest test suite covering the install behavior against fixture projects (assertions on resulting files, not actual server boot).
- Documentation: README usage section and a docstring on the install task module.

### Out of scope

- SSL / HTTPS configuration paths.
- Anything that depends on Ecto, mailers, or assets requiring a build step (the targeted `mix phx.new` flags are `--no-ecto --no-email`).
- A separate `mix phx.lite.serve` task (open question â€” defer).
- Elixir releases (open question â€” defer).
- Hardening for public-internet, multi-tenant, or production-grade deployments.
- Editing `config/prod.exs` (open question â€” defer until we have evidence it's needed).

## Architecture & Design Decisions

1. **Templated replacement, not in-place editing.**
   The spec leaves the strategy open. Recommend **wholesale replacement** of `config/runtime.exs` with a templated file shipped in `priv/templates/`. Rationale:
   - The set of edits needed (SSL, host, secret) touches enough sites that surgical edits via Igniter's config helpers / Sourceror would be brittle across Phoenix versions.
   - A templated file is easier to read, review, and reason about for the developer who is shipping it.
   - The primary downside (drift from new Phoenix scaffolding) is acceptable for a tool aimed at internal apps and prototypes, and is mitigated by the backup of the original.

2. **Module + Mix task naming.**
   Follow the established Igniter convention (see `Mix.Tasks.GitOps.Install` in `deps/git_ops/`):
   - Task module: `Mix.Tasks.PhoenixLite.Install` (invoked as `mix phoenix_lite.install` and resolved by `--install phoenix_lite`).
   - Companion `Mix.Tasks.PhoenixLite.Install.Docs` module providing `short_doc/0`, `example/0`, `long_doc/0`.
   - Wrap the task definition in `if Code.ensure_loaded?(Igniter) do ... end` so the package degrades cleanly if Igniter isn't present.

3. **Backup location.**
   Before overwriting `config/runtime.exs`, copy the original to `config/runtime.exs.phoenix_lite.bak` in the host project. Rationale:
   - Predictable, co-located with the file it backs up â€” easy to discover.
   - The `.phoenix_lite.bak` suffix prevents collisions with editor swap files and makes the source of the backup obvious.

4. **Idempotency.**
   On a second run, detect that the templated `runtime.exs` is already in place (e.g. via a sentinel comment in the template such as `# Generated by phoenix_lite â€” do not hand-edit`) and skip both the overwrite and the backup creation. This avoids the bug where re-running the installer overwrites a real user-edit backup with the templated content.

5. **Where the prod-required values come from.**
   - `secret_key_base`: generate at install time via `:crypto.strong_rand_bytes(48) |> Base.encode64()` and bake the literal string into the templated `runtime.exs`. This satisfies "no env var required" while avoiding a hard-coded, known-to-the-world key.
   - Host: literal `"0.0.0.0"`.
   - Port: read from `PORT` env at runtime with a sensible default (e.g. `4000`).
   - `server: true` set unconditionally so `mix phx.server` boots the endpoint in prod.

6. **Igniter API surface.**
   - Use `Igniter.create_or_update_file/4` (or `Igniter.copy_template/4` if available) for both the backup write and the templated replacement.
   - Use `Igniter.add_notice/2` (or the equivalent) to surface the file-change summary.
   - Use `Igniter.Project.Config.configure_new/4` only if we end up needing to nudge `config.exs` or `config/prod.exs` (likely not in v1).

7. **Public module.**
   `lib/phoenix_lite.ex` stays as a thin namespace module. The install task is the only behavior in v1; no need for a separate `PhoenixLite.Installer` module unless logic grows.

## Implementation Steps

1. **Promote `:igniter` to a runtime dep.**
   - Files: `mix.exs`
   - Details: `:igniter` is currently `only: [:dev, :test]`. Move it to a normal dep so the install task can rely on `Igniter.Mix.Task` at install time on a user's machine. Run `mix deps.get` afterward.

2. **Create the templated `runtime.exs`.**
   - Files: `priv/templates/runtime.exs.eex` (new)
   - Details: Author the templated `runtime.exs` an installed app will receive. Includes:
     - The `# Generated by phoenix_lite â€” do not hand-edit` sentinel comment at the top.
     - `import Config` and the `if config_env() == :prod do ... end` block.
     - `config :<%= @otp_app %>, <%= @endpoint_module %>,` with `http: [ip: {0, 0, 0, 0}, port: ...]`, `url: [host: "0.0.0.0", port: ...]`, `server: true`, `secret_key_base: "<%= @secret_key_base %>"`.
     - Explicit absence of any `https:` block.
   - Use EEx assigns rather than module-name guessing so the same template works for any `mix phx.new` output.

3. **Add a helper that introspects the host project.**
   - Files: `lib/phoenix_lite/host_project.ex` (new)
   - Details: Small module with functions to:
     - Read the host project's `:app` (otp_app) from its `mix.exs` via `Igniter.Project.Application` or by parsing `Mix.Project.config/0` of the target.
     - Compute the endpoint module name (`<App>Web.Endpoint`) â€” same convention `phx.new` produces.
     - Detect "is this a Phoenix project?" by checking for `phoenix` in the host project's deps. If not, return `{:error, reason}` so the task can fail clearly.
   - Keep this module pure / data-returning â€” no `Igniter` writes here. Easier to test.

4. **Implement the Mix task.**
   - Files: `lib/mix/tasks/phoenix_lite.install.ex` (new)
   - Details: Mirror the structure of `Mix.Tasks.GitOps.Install`:
     - `Mix.Tasks.PhoenixLite.Install.Docs` with `short_doc/0`, `example/0`, `long_doc/0`.
     - `Mix.Tasks.PhoenixLite.Install` guarded by `if Code.ensure_loaded?(Igniter) do`.
     - `use Igniter.Mix.Task`.
     - `info/2`: `group: :phoenix_lite`, `example: "mix igniter.install phoenix_lite"`, `schema: [force: :boolean]`, `defaults: [force: false]`, `only: nil` (the package itself is runtime â€” but the task is dev-time only; `only: [:dev]`).
     - `igniter/1`:
       1. Bail out with a clear error notice if the host project isn't Phoenix.
       2. If `config/runtime.exs` already contains the sentinel and `--force` was not passed, add a notice ("already installed; pass --force to overwrite") and return the igniter unchanged.
       3. Otherwise: read the existing `config/runtime.exs`, write it to `config/runtime.exs.phoenix_lite.bak` (skip if backup already exists), generate a fresh `secret_key_base`, render the template with `otp_app`, `endpoint_module`, and `secret_key_base` assigns, write it to `config/runtime.exs`.
       4. Add a notice listing the files written and backed up.

5. **Wire the template into the package.**
   - Files: `mix.exs` (touch `package` metadata), the new `priv/templates/runtime.exs.eex`
   - Details: Ensure `priv/` is included in the Hex package once we publish (when a `package` keyword is added later). Not strictly required for path-dep / local install. Note in code, defer the `package` block until publish.

6. **Tests against fixture projects.**
   - Files: `test/phoenix_lite/install_test.exs` (new), `test/support/fixtures/...` or use `Igniter.Test`
   - Details: Igniter ships `Igniter.Test` helpers for running tasks against synthetic projects. Use those (avoids needing a real `phx.new` checkout). Cover:
     - Template is written with the expected host, port, `server: true`, and a non-empty `secret_key_base`.
     - Backup file is created at `config/runtime.exs.phoenix_lite.bak` containing the original content.
     - Second run without `--force` is a no-op (no overwrite, no second backup).
     - Second run with `--force` overwrites again but does not clobber an existing backup.
     - Running against a non-Phoenix project produces a clear error notice and writes nothing.
   - One minimal `test_helper.exs` adjustment if Igniter test helpers need it.

7. **Update README.**
   - Files: `README.md`
   - Details: Replace the placeholder `**TODO: Add description**` and the stub `def deps` block with a real description and the documented install command:
     ```
     mix igniter.new myapp --with phx.new --with-args="--no-ecto --no-email" --install phoenix_lite
     ```
   - Document what the installer changes, where the backup goes, and the `--force` flag.
   - Note: `git_ops` is configured with `manage_readme_version: true` â€” keep the version reference exactly where the existing README has it so `mix git_ops.release` continues to find and bump it.

8. **Conventional-commit history.**

## Dependencies & Ordering

- **Step 1 (promote `:igniter`) must come first** â€” every later step compiles against `Igniter.Mix.Task`. Without this, the task module won't load on a fresh install.
- **Step 3 (host_project helper) must come before Step 4 (task)** â€” the task calls into it.
- **Step 2 (template file) and Step 3 (helper) are independent** and can be done in either order.
- **Step 6 (tests) depends on Steps 2â€“4.**
- **Step 7 (README) and Step 8 (commits) are presentation-layer and can happen alongside Step 6.**

## Edge Cases & Risks

- **Host project already hand-edited `runtime.exs`:** the backup-then-overwrite flow preserves their work, but they'll need to re-apply edits on top of the templated file. Mitigation: clear notice telling them where the backup lives.
- **Backup already exists from a prior run:** do **not** overwrite it. A prior backup represents the user's actual original; a second run's "original" is our generated file and is worthless as a backup. Skip the backup write if the path already exists.
- **Igniter not loaded at compile time:** the `Code.ensure_loaded?(Igniter)` guard avoids a load-time crash; behavior matches the `git_ops` reference task.
- **Template drift across Phoenix versions:** new `phx.new` versions may rely on additional `runtime.exs` config (e.g. clustering, telemetry endpoints). Mitigation: keep the template minimal â€” only what's needed to boot â€” and revisit as Phoenix evolves.
- **Generated `secret_key_base` baked into source:** by design, this gets committed if the developer commits their app. That's acceptable for the targeted use cases (internal / SSG / prototype) per the spec's "Out of Scope" hardening note. Document this trade-off in the README.
- **Non-Phoenix project:** detection via deps inspection. If the heuristic is wrong (e.g. an umbrella with Phoenix in a sub-app), surface a clear notice rather than silently doing nothing.
- **`--no-ecto --no-email` flag combinations:** the template should make no assumption that `Ecto.Repo` or a mailer module exists. Avoid referencing them in the templated `runtime.exs`.

## Testing Strategy

- **Igniter.Test-based unit tests** in `test/phoenix_lite/install_test.exs` covering each acceptance criterion in the spec.
- **No live `mix phx.server` boot** in CI â€” assert on the resulting file contents (host, port, `server: true`, presence of `secret_key_base`, presence of the sentinel comment) rather than starting a real server.
- **Manual smoke test** before tagging a release:
  ```
  mix igniter.new smoketest --with phx.new --with-args="--no-ecto --no-email" --install phoenix_lite
  cd smoketest && mix deps.get && MIX_ENV=prod mix phx.server
  ```
  Confirm it serves on `http://0.0.0.0:4000`. This is the spec's primary acceptance criterion and cannot be fully exercised by unit tests.
- **Re-run idempotency check** as part of the smoke test: run the installer a second time on the same project and confirm the notice + no file diff.
